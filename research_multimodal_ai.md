# 多模态AI vs 纯规则生成：光学画图题小程序技术方案调研报告

**调研日期**：2026-06-14  
**调研目标**：评估"集成多模态AI API批改手绘光路图" vs "纯规则生成+Canvas交互"两种方案的可行性，为浙工大光学刷题小程序选择最优技术路线。  
**数据来源**：网络公开资料（2024-2026）、学术基准测试结果、厂商API定价文档、项目已有数据资产。

---

## 一、当前多模态AI（视觉+文本）能力现状

### 1.1 多模态大模型在"看图理解"方面的进展

2024-2026年，多模态大语言模型（MLLM）经历了从"拼接式"到"原生多模态"的架构演进。代表模型包括：

| 模型 | 开发者 | 多模态输入 | 视觉理解定位 |
|------|--------|-----------|-------------|
| GPT-4o / GPT-4.1 | OpenAI | 图像+文本 | 通用场景理解强，但几何推理有限 |
| Claude 3.5/4.6 Sonnet | Anthropic | 图像+文本 | 文档、图表理解好，无语音/视频 |
| Gemini 2.5 Pro/Flash | Google | 图像+文本+语音+视频 | 原生多模态，多模态能力强 |
| Qwen2.5-VL / Qwen-Plus | 阿里云 | 图像+文本+语音 | 视觉Agent、结构化数据提取 |
| 文心4.5 / X1 | 百度 | 图像+文本 | 中文视觉理解、多工具调用 |
| GLM-4.5V / GLM-4.1V-9B | 智谱AI | 图像+文本 | 3D空间推理、高效开源 |
| DeepSeek-V3.2 | 深度求索 | 图像+文本 | 代码+数学推理，多模态非核心优势 |

这些模型在日常图像识别（如OCR、物体分类、场景描述）上已接近商用水平，但在**精细几何理解**方面存在系统性短板。

### 1.2 关键基准测试：多模态AI在几何/物理图形推理上的真实表现

本次调研重点关注了两个与"光学画图题批改"高度相关的学术基准测试：

#### （1）RBench-V（视觉推理基准，2025-05）

由清华大学、腾讯混元、斯坦福、CMU联合发布，专门测试模型**"基于图像进行推理并输出图形操作"**的能力（如画辅助线、描点连线、绘制光路、标注区域）。

| 模型 | 整体准确率 | 人类专家水平 |
|------|-----------|-------------|
| o3 (OpenAI最强推理模型) | **25.8%** | 82.3% |
| Gemini 2.5 | 20.2% | 82.3% |
| GPT-4o | ~15-20% | 82.3% |
| Qwen2.5-VL / InternVL / LLaVA | 8-10% | 82.3% |

**核心发现**：
- 即使最强的o3，准确率也仅人类的 **1/3**。
- 开源模型在部分维度接近"随机作答"水平。
- 模型倾向于**"用文字绕过图形"**——将几何问题抽象为坐标代数表达，而非真正理解图像中的空间关系。
- **结论**：当前多模态AI在"看懂了图并给出精确几何判断"这一任务上，远未达到可靠商用水平。

#### （2）MV-MATH（多图数学推理基准，2025-03，CVPR 2025）

中科院自动化所发布，测试多图数学推理。

| 模型 | 总体准确率 | 算术 | 组合几何 |
|------|-----------|------|---------|
| Claude-3.5 | 33.9% | 54.2% | **27.0%** |
| GPT-4o | 32.1% | - | - |
| Gemini-1.5-Pro | 29.1% | - | - |
| Qwen-vl-max | 26.9% | - | - |
| 人类水平 | 76.5% | - | - |

**核心发现**：
- 在**组合几何**（与光学光路图高度相似的"图形+推理"任务）上，最强模型仅 **27%** 准确率。
- 在困难问题上，Claude-3.5暴跌至 **26.6%**。
- **图像序列输入优于合并输入**，说明结构化视觉信息很重要——但即便如此，表现仍不佳。

#### （3）通用视觉理解的局限性（2025-2026综述）

UNC Chapel Hill、KAIST等研究团队指出，多模态大模型在以下方面存在系统性不足：
- **方向感困惑**：难以识别图像旋转角度。
- **细节感知有限**：对小物体、细线条（如光路图中的虚线、焦点标记）识别不足。
- **空间关系理解不准确**：难以判断"两条线是否相交于某点"、"某点是否在某线上"。
- **视觉幻觉**：有时会"看到"图像中不存在的内容。
- **几何关系推理困难**：难以进行多步骤几何推导。

**对光学光路图的具体影响**：
光学画图题的核心判断点恰恰是这些模型的短板：
1. "这条光线是否**平行于光轴**？"（方向感+空间关系）
2. "折射后的光线是否**经过像方焦点F'**？"（细节感知：F'是一个小标记点）
3. "两条出射光线的**交点**是否落在正确位置？"（空间关系+推理）
4. "物距与焦距的比例关系"（几何测量+推理）

## 二、微信小程序/Web可集成的多模态API调研

### 2.1 技术架构模式

微信小程序无法直接从前端调用外部AI API（域名限制+安全），标准架构为：

```
小程序端：拍照/上传图片 → wx.compressImage压缩 → 上传至业务服务器
业务服务器：接收图片 → Base64编码或转URL → 组装Prompt → 调用大模型API → 解析返回 → 返回给小程序
```

**关键工程注意事项**：
- **图片压缩**：大模型按Token计费，分辨率过高会大幅增加费用和延迟。建议压缩至1080p以内。
- **延迟**：端到端通常 **1-5秒**（含网络传输+模型推理）。
- **网络依赖**：小程序必须联网，无法离线使用。

## 三、项目已有数据资产分析

### 3.1 画图题数据现状

根据项目已提取的 `extracted_drawings.json` 和 `extracted_images/`：

- **已提取图片**：18张（来自2份PDF，各9页）
- **标准作图题**：前6页为教材扫描页，后3页为手写总结
- **第6页（最清晰，蓝色打印版）**包含 **6道标准小题**：
  1. 作轴上实物点A的像A'
  2. 作轴上虚物点A的像A'
  3. 作垂轴实物AB的像A'B'
  4. 作垂轴虚物AB的像A'B'
  5. 画出焦点F的位置
  6. 画出焦点F'的位置
- **第7-9页**：手写总结，系统汇总了 **9种物距** 下的成像规律（`l = -∞, -2f, -f, -f/2, 0, f/2, f, 2f, ∞`）

### 3.2 画图题的核心特征：类型有限、规则确定

这是本次调研的**最关键发现**：

| 特征 | 说明 | 对技术方案的影响 |
|------|------|-----------------|
| **题型有限** | 仅6-8种标准类型 | 规则库可以穷举覆盖 |
| **规则固定** | 每类题的标准光路规则是确定的 | 程序化判断逻辑可100%精确 |
| **几何约束明确** | 平行于光轴→过F'；过F→平行于光轴；过节点→方向不变 | 每条光线可用向量/斜率精确验证 |
| **评价标准客观** | 光线是否经过某点、是否平行、是否共轭 | 不存在主观评分，适合规则引擎 |
| **交互步骤可拆解** | 每道题可拆为3-5个标准步骤 | 支持步骤级纠错和引导 |

**核心规则示例**（理想光组成像）：
- 规则1：从物点发出**平行于光轴**的光线，经理想光组后**必过像方焦点F'**。
- 规则2：从物点发出**过物方焦点F**的光线，经理想光组后**必平行于光轴**。
- 规则3：从物点发出**过物方节点J**的光线，经理想光组后**必过像方节点J'且方向不变**。
- 规则4：从物点发出**过物方主点H**的光线，经理想光组后**必过像方主点H'且高度不变**。
- 规则5：实像=实际光线交点；虚像=光线反向延长线交点。

这些规则是**初等几何+光学定律**的直接推论，没有模糊地带，不需要"理解"，只需要"验证"。

## 四、方案对比：多模态AI API vs 纯规则生成

### 4.1 方案A：集成多模态AI API

**架构**：用户手绘光路图 → 拍照上传 → 后端调用Qwen-VL/GPT-4o等API → AI返回批改结果

**优点**：
- 开发速度快：无需编写光学规则，只需设计Prompt和解析结果。
- 通用性强：理论上可以处理任何手绘图，不限于预设题型。
- 体验自然：用户可以在纸上画，拍照上传即可。

**缺点**：
- **准确率不可靠**：
  - RBench-V显示最强模型在视觉几何推理上仅25.8%准确率。
  - 手绘图的线条不直、标记不清、拍照角度倾斜，会进一步降低识别率。
  - 模型可能"幻觉"出不存在的光线或误判焦点位置。
- **成本持续**：
  - 每批改一张图需调用API，按量计费。
  - 若用户量增长，成本线性增长。
  - 国内模型约0.01-0.1元/次，国际模型约0.05-0.2元/次。
- **延迟高**：
  - 端到端1-5秒，无法实时反馈。
  - 用户画完一条线后无法"即时"判断对错，必须等整图上传后统一批改。
- **离线不可用**：必须联网，电梯/地铁/校园网差时无法使用。
- **难以步骤级纠错**：
  - 多模态AI通常返回整图的整体评价，难以精确指出"第2步的光线应该过F'而不是F"。
  - 实现步骤级拆解需要额外的Prompt工程和结果解析，进一步增加成本和延迟。
- **黑盒不可控**：模型更新可能改变行为，无法保证规则一致性。

### 4.2 方案B：纯规则生成（前端Canvas + 后端规则验证）

**架构**：小程序内嵌Canvas → 用户在屏幕上画光线（系统提供光轴/主面/焦点参考点）→ 前端记录每根线的起点/终点/斜率 → 后端用几何规则实时验证 → 即时反馈对错

**优点**：
- **准确率100%**：
  - 基于初等几何规则（斜率、交点、平行判断）精确验证。
  - 不存在"幻觉"或"模糊理解"，对错判断完全确定。
- **延迟极低**：
  - 前端本地即可完成大部分几何计算（毫秒级）。
  - 用户画完一条线后，**即时**得到反馈（"正确，这条线确实过F'"或"错误，请检查是否经过像方焦点"）。
- **零边际成本**：
  - 无API调用费用，无论多少用户、多少次练习，服务器成本几乎不变（仅计算资源）。
- **离线可用**：
  - 规则验证逻辑可放在前端（微信小程序支持本地计算），无需网络即可练习。
- **步骤级精确纠错**：
  - 系统知道用户当前在做第几步，知道该步的标准规则是什么。
  - 可以给出精确反馈："步骤2要求：平行于光轴的光线折射后必须过像方焦点F'。你的光线经过了F，请区分F（物方）和F'（像方）。"
- **可扩展性**：
  - 新增题型只需增加规则配置，无需重新训练或调优模型。
  - 可以积累用户画图数据，后续用于训练专门的视觉模型（远期）。

**缺点**：
- **开发成本高**：
  - 需要将6-8类题型的标准光路规则全部数字化（几何约束表达式）。
  - 需要实现Canvas交互和手势识别（画直线、擦除、提交）。
  - 需要设计"步骤拆解"和"猜测错因"的逻辑（如用户画错是因为混淆了F和F'，还是混淆了主面位置）。
- **用户体验限制**：
  - 用户必须在小程内画图，不能像纸上那样自由。
  - 需要适应Canvas的触屏操作（对小屏手机可能不够精细）。
  - 系统需要提供"吸附"功能（自动对齐光轴、焦点），否则手绘直线很难精确。
- **题型覆盖有限**：
  - 只能处理已规则化的题型，若用户遇到未配置的题型，无法支持。
  - 但本项目题型恰好是有限且确定的（6-8种），此缺点不致命。
- **无法识别教材截图**：
  - 不能处理"用户上传一张教材上的图，系统解析题意"的场景（但这不是核心需求，核心需求是"用户画图并获反馈"）。

### 4.3 对比总表

| 维度 | 方案A：多模态AI API | 方案B：纯规则生成 |
|------|-------------------|-----------------|
| **开发周期** | 短（1-2周） | 中（4-8周） |
| **开发成本** | 低（Prompt工程+解析） | 中高（规则库+Canvas+交互） |
| **运行成本** | 持续按量计费（0.01-0.2元/次） | 几乎为零（本地计算） |
| **延迟** | 1-5秒/次 | 毫秒级（实时） |
| **准确率** | 25-35%（几何推理） | 100%（确定性规则） |
| **离线可用** | 否 | 是（可前端本地计算） |
| **步骤级纠错** | 困难（黑盒） | 精确（白盒规则） |
| **用户体验** | 拍照上传，等结果 | 实时画线，即时反馈 |
| **可扩展性** | 依赖API能力边界 | 规则配置即可扩展 |
| **长期可控性** | 低（模型更新不可控） | 高（自有逻辑） |
| **适合本题库** | 不适合（题型有限、规则确定） | **高度适合** |

## 五、结论与推荐方案

### 5.1 核心结论

**多模态AI API目前不适合作为"光学画图题实时纠错"的核心引擎**，原因有三：

1. **准确率不足**：当前最强模型在视觉几何推理任务上仅25-35%准确率，而光学画图题的批改要求100%精确（光线是否过焦点、是否平行，是刚性几何约束）。
2. **延迟与成本不适合实时交互**：API调用需要1-5秒且按次计费，而画图题的理想交互是"画一条线→即时反馈→修正→下一条线"。
3. **本题库的特殊性使多模态AI成为"杀鸡用牛刀"**：题型仅6-8种，规则完全确定，用规则引擎比用通用AI更精确、更可控、更便宜。

### 5.2 推荐方案：方案B（纯规则生成）为主，方案A为辅

**主路径：方案B — 纯规则生成（Canvas+规则验证）**

- 在微信小程序内使用Canvas 2D提供画图界面。
- 系统预设光轴、主面H/H'、焦点F/F'的参考位置（用户不可移动，作为题目条件）。
- 用户用手指画光线，前端记录每条线段的起点/终点。
- 根据题型调用对应规则引擎，实时验证：
  - 斜率是否≈0（平行于光轴）？
  - 是否经过某点（F'或H'）？
  - 两条线交点位置是否正确？
- 支持步骤级拆解：每道题分为3-5步，用户逐步完成，每步即时反馈。
- 答错时给出针对性诊断："你画的光线经过了F而非F'，请复习物方焦点与像方焦点的区别"。

**辅助：方案A — 多模态API的有限使用**

多模态AI并非完全无用，可在以下场景作为**辅助功能**：
- **题目录入**：用户上传教材/真题截图，用多模态API（如Qwen-VL或文心4.5）提取题意文字和光路条件，辅助人工录入题库。
- **图片搜索**：用户上传手写笔记照片，API识别内容并推荐相关知识点卡片。
- **通用答疑**：用户拍摄教材上的任意题目（非画图题），API提供解析（这类任务对准确率要求不如画图题严格）。

**不推荐**：用多模态API直接批改用户的手绘光路图。这是当前技术能力边界之外的任务。

### 5.3 技术实施路线图建议

| 阶段 | 时间 | 目标 | 技术要点 |
|------|------|------|---------|
| **Phase 1** | 2-3周 | 规则库+MVP Canvas | 实现6类标准题型的规则验证逻辑；基础Canvas画线（自动吸附光轴/焦点） |
| **Phase 2** | 2-3周 | 步骤级交互+诊断 | 每题拆3-5步；即时反馈；错因猜测（混淆F/F'、混淆H/H'、实像虚像判断错误） |
| **Phase 3** | 1-2周 | 针对性练习+数据积累 | 连续错某步骤则推送专项练习；积累用户画图轨迹数据 |
| **Phase 4** | 远期 | 多模态辅助功能 | 接入Qwen-VL/文心4.5实现"拍照搜题"、"笔记识别"等辅助功能 |

### 5.4 风险提示

1. **Canvas触屏体验**：手机屏幕小，手指画直线不够精确。必须实现：
   - 吸附对齐（靠近光轴时自动吸附到水平）
   - 两点确认（点起点→点终点，而非拖拽）
   - 撤销/重做
2. **规则库完备性**：初期只覆盖6类标准题，后续扩展组合系统（F'/H'求解）时需补充新规则。
3. **多模态API价格波动**：若远期使用多模态API辅助功能，需关注国内厂商价格战（文心4.5已降至GPT-4.5的1%价格）。
